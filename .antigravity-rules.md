# Reglas de Proyecto: Lumina App

Estas instrucciones son de lectura obligatoria para cualquier agente IA (Antigravity u otros) trabajando en este proyecto.

## 🎨 1. Estilizado y CSS en Svelte
*   **Regla de Oro:** **NUNCA** utilices estilos en línea (`style="..."`) ni bloques `<style>` dentro de los componentes `.svelte` a menos que sea estrictamente necesario para estilos dinámicos muy complejos que dependan de JavaScript (e.g. posiciones calculadas en tiempo de ejecución o variables CSS inline como `--progress: 50%`).
*   **Archivos CSS Separados:** Todos los estilos deben escribirse en archivos `.css` independientes. Si el proyecto tiene una arquitectura de estilos (por ejemplo, en `src/app.css` o dentro de una carpeta `src/styles/`), se deben añadir las clases allí.
*   **Clean Svelte:** Un componente `.svelte` debe contener principalmente `<script>` y `<markup>`. Mantén la lógica de apariencia completamente separada en CSS puro para fomentar la reutilización y limpieza del código en este proyecto Tauri.

## 🧠 3. Separación de Lógica y UI
* **Estado y Backend fuera de Svelte:** Al igual que el CSS, la lógica densa (comunicación con Tauri usando `invoke`, manejo de estados complejos con Runes o Svelte Stores) NO debe ir dentro del componente `<script>` de `archivo.svelte`.
* **Clases State Separadas:** Debes crear un archivo `.svelte.ts` (e.g., `SalaryHistoryState.svelte.ts`) en la carpeta `src/lib/logic/` o `src/lib/stores/`.
* **Clean Components:** El componente Svelte solo debe instanciar su clase "State", pasarle las Props, y dedicarse de forma exclusiva a iterar y pintar variables reactivas del State (`state.isLoading`, `state.history`). No más funciones asíncronas de base de datos colgadas directamente en la vista.
